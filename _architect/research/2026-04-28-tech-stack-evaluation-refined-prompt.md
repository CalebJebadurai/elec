# Tech Stack Evaluation: Performance, Scalability, and Cost Optimization

**Date:** 2026-04-28  
**Context:** Post-strategic analysis reassessment  
**Type:** Technical architecture evaluation  

---

## Problem Statement

The elec platform currently operates on a Python FastAPI backend with asyncpg for PostgreSQL access, a React 19 frontend built with Vite 8, and PostgreSQL 16 as the data store, all deployed on free-tier infrastructure (Railway for backend, Vercel for frontend). The April 27 strategic analysis recommended keeping this stack and improving it incrementally through Redis caching, database query optimization, test coverage, and monitoring additions. However, this recommendation was made within the context of a broader platform improvement initiative focused on monetization and growth, not as a dedicated tech stack evaluation against alternatives. The question now being raised is whether the current Python/React/PostgreSQL stack is fundamentally the right technical foundation for the platform's performance requirements, scalability targets, and cost constraints, or whether a different technology choice would better serve these objectives.

## Current State

The backend runs FastAPI with uvicorn using asyncpg for raw SQL queries against PostgreSQL, with no ORM layer. As observed in [api/main.py](api/main.py), [api/routes.py](api/routes.py), and [api/national_routes.py](api/national_routes.py), the application makes extensive use of parameterized queries with asyncpg's connection pooling. Database access patterns include single-row lookups by ID, filtered result sets with pagination (up to 500 records per page), complex aggregation queries using CTEs for national-level statistics, and historical time-series queries joining on year and constituency. The database schema in [init.sql](init.sql) contains a single tcpd_ae table with 574,000 rows and 48 columns, with composite indexes on state_name + year, state_name + constituency_name + year, year + position, and trigram indexes for candidate name search. National aggregation endpoints execute multi-CTE queries that require full-table scans filtered by position or election type, which can take two to five seconds on a cold start according to the April 27 research findings.

The caching layer has recently been upgraded from purely in-memory Python dictionaries to Redis-backed caching with in-memory fallback, as seen in [api/cache.py](api/cache.py). Response caching uses a five-minute TTL for most endpoints, with a one-hour TTL for national statistics. Rate limiting remains in-memory using Python dict structures in [api/main.py](api/main.py) lines 39-55, which means it cannot enforce limits correctly across multiple server instances — each process maintains its own independent counter. The current Railway deployment configuration in [railway.toml](railway.toml) runs a single instance (numReplicas = 1), so this limitation has not manifested as a production issue yet.

The frontend is React 19 with Vite 8, using Firebase for phone authentication and JWT tokens stored in localStorage for API access. The application makes synchronous API calls to FastAPI endpoints and performs client-side prediction calculations, meaning the prediction simulator workload is entirely offloaded to the browser. Static assets are served via Vercel's CDN. As documented in [frontend/package.json](frontend/package.json), the build produces an optimized production bundle that Vercel deploys globally.

The current deployment architecture runs entirely on free tiers: Railway's Hobby plan with 500 hours per month of runtime and five hundred megabytes of RAM, Vercel's free plan with one hundred gigabytes of bandwidth per month, and a managed PostgreSQL instance. The database connection in [api/database.py](api/database.py) uses asyncpg connection pooling with a configurable pool size (default two to ten connections) and disables SSL certificate verification for Railway's self-signed certificates by setting check_hostname to False and verify_mode to CERT_NONE. The April 27 technical debt assessment identified this as a security concern but noted it was necessary for Railway deployment.

The existing [docker-compose.yml](docker-compose.yml) and [infra/terraform/](infra/terraform/) directory demonstrate that a self-hosted deployment architecture has been designed but not activated. The Terraform configuration provisions a three-tier GCP deployment with separate compute instances for the API, dedicated Cloud SQL for PostgreSQL, and Cloud CDN for static assets, but this has never been deployed due to cost considerations.

Performance characteristics have not been formally measured in production. There is no APM monitoring, no p50/p95/p99 latency tracking, and no throughput measurement. The April 27 research document noted that complex national aggregation queries could take two to five seconds on cold starts, and the strategic plan set a target of sub-500ms median response time for the top ten endpoints with 95th percentile under two seconds. No baseline measurements exist to confirm whether these targets are currently being met or missed. The only observability is basic stdout logging and Sentry error tracking when configured.

## Desired End State

The platform's tech stack should demonstrably support the performance, scalability, and cost requirements of a monetized analytics service while maximizing developer productivity for a solo developer. Specifically, the stack must be capable of serving API requests with median latency below five hundred milliseconds and 95th percentile below two seconds during peak traffic periods when handling concurrent users in the thousands. The system must scale horizontally without requiring architectural rework, meaning stateless request handling with external state management for rate limiting, session storage, and response caching. The infrastructure cost for serving ten thousand monthly active users with an estimated fifty API requests per session should remain under one hundred dollars per month when operating on paid hosting tiers. The developer must be able to implement new features, fix bugs, and deploy changes with confidence, meaning the stack's ecosystem should provide mature testing tools, established deployment patterns, and a large community for troubleshooting. When a performance bottleneck is identified, it should be clear whether the bottleneck is inherent to the technology choice or simply a matter of optimization, and the path to resolution should be well-documented within the language and framework's ecosystem.

The outcome is not necessarily a different tech stack, but rather a documented, evidence-based justification for either keeping the current stack or migrating to specific alternatives, with explicit trade-off analysis covering performance characteristics, scaling patterns, cost models, and migration effort.

## Scope Definition

This evaluation is scoped to compare the current Python FastAPI + React + PostgreSQL stack against plausible alternatives that could realistically be implemented by a solo developer within three to six months while maintaining feature parity. Comparisons should include at minimum: (1) the current stack optimized with Redis caching, connection pooling tuning, database query optimization, and horizontal scaling on paid hosting; (2) a Go-based backend with Gin or Echo framework replacing FastAPI while keeping React frontend and PostgreSQL database; (3) a Node.js backend with Express or Fastify replacing FastAPI while keeping React frontend and PostgreSQL; (4) replacement of PostgreSQL with a different database technology such as MongoDB or DynamoDB if there is evidence that the relational model is a bottleneck; (5) replacement of the current free-tier hosting (Railway + Vercel) with alternative platforms such as Fly.io, Render, DigitalOcean App Platform, or self-hosted GCP using the existing Terraform configuration, even if the backend language remains Python.

The evaluation must consider the specific workload characteristics of this application: read-heavy with a 95%+ read-to-write ratio, complex analytical queries requiring aggregation across hundreds of thousands of rows, infrequent writes limited to user bookmarks and prediction saves, client-side prediction computation that offloads the most CPU-intensive work from the backend, and traffic patterns that spike dramatically during election periods (potentially 10-50x baseline) then drop to near-zero between elections.

Scope explicitly excludes: (1) trendy but immature stacks that lack production track records such as Bun, Deno Deploy, or Rust-based web frameworks like Actix that would require the solo developer to learn a new language; (2) serverless-first architectures using AWS Lambda or Google Cloud Functions that would require rewriting the entire request handling model and introduce cold start penalties that conflict with the sub-500ms latency target; (3) microservices decomposition — the application is small enough that a monolithic backend is appropriate; (4) frontend framework replacement — React is not the bottleneck, and rewriting the frontend in Vue, Svelte, or a meta-framework like Next.js provides no performance or cost benefit; (5) bleeding-edge database technologies like Clickhouse, TimescaleDB, or graph databases that would require complete schema redesign and data migration without clear evidence of benefit for this workload.

The evaluation must include cost modeling at three traffic levels: baseline (one thousand MAU, fifty thousand monthly API requests), moderate growth (ten thousand MAU, five hundred thousand monthly API requests), and election-season peak (fifty thousand concurrent users, five million API requests over a three-day period). Cost modeling must account for compute, database, bandwidth, monitoring, and any technology-specific costs such as Firebase phone authentication pricing.

## Constraints and Assumptions

This evaluation operates under the following constraints and assumptions, each stated explicitly so they can be challenged or adjusted during the analysis.

**Developer context:** A solo developer working full-time on this project, with strong Python and React skills developed through building the current platform. The developer has moderate familiarity with Node.js and basic familiarity with Go but has never built a production application in Go. Any migration to a new backend language carries a learning curve penalty that must be accounted for in timeline estimates. The opportunity cost of migration time is high — three months spent rewriting the backend in Go is three months not spent building monetization features, expanding data coverage, or improving the product.

**Data characteristics:** The dataset is historical election results totaling 574,000 rows with 48 columns per row. The data grows slowly — approximately ten to thirty thousand new rows per year as new elections occur and results are ingested. The data is write-once read-many, meaning historical election results never change once recorded. The only mutable data in the system is user accounts, bookmarks, and API keys, which represent a tiny fraction of total data volume. The database is small enough to fit comfortably in memory on modern hardware, meaning a read-optimized caching strategy should be highly effective regardless of the underlying database technology.

**Query patterns:** The most frequent queries are single-state, single-year result sets returning hundreds to thousands of rows, which currently use indexed lookups on state_name + year. The second most frequent pattern is constituency detail views that fetch all election results for a single constituency across all years, using the state_name + constituency_name + year composite index. The most expensive queries are national aggregations that compute party-level or state-level statistics across all years, which currently use CTEs and full-table scans. These expensive queries are cached aggressively with one-hour TTLs, so they are only computed once per hour even during high traffic periods.

**Traffic patterns:** The platform experiences extreme cyclicality aligned with Indian election schedules. During election months (typically April-May for state elections, with staggered elections throughout the year), traffic can spike to ten to fifty times baseline levels as users run prediction scenarios and share results. Between elections, traffic drops to a baseline of political researchers, journalists preparing stories, and hobbyist election analysts. This cyclicality has two implications: (1) infrastructure must handle peak load, but paying for peak capacity year-round is economically wasteful, making auto-scaling critical; (2) cold start penalties are tolerable during low-traffic periods but unacceptable during election season when users expect instant responsiveness.

**Cost sensitivity:** The platform must be financially sustainable as an independent product, meaning infrastructure costs must remain below twenty percent of revenue at scale. Given the target of one thousand dollars monthly recurring revenue within twelve months, infrastructure costs should not exceed two hundred dollars per month at ten thousand MAU. Free-tier hosting is acceptable during initial development but is not viable long-term due to lack of SLAs, resource limits, and risk of service disruption during traffic spikes. The cost model must account for Indian pricing — Firebase phone authentication costs scale with SMS volume, and Indian SMS rates are among the lowest globally (approximately 0.02 USD per message), but verification volume during election season could still be substantial.

**Migration risk tolerance:** Any tech stack change must be implementable without a full service outage. The platform is live with active users, meaning a big-bang rewrite and cutover is unacceptable. A migration approach must allow for incremental replacement (e.g., running old and new backends side-by-side, gradually migrating endpoints) or must be completable during a planned maintenance window of no more than four hours. Data migration from PostgreSQL to a different database technology must be validated with checksums or test queries to ensure no data loss or corruption.

**Ecosystem maturity requirement:** Any alternative technology must have mature, production-grade libraries for the platform's critical dependencies: PostgreSQL connection pooling (or equivalent for alternative databases), JWT authentication and verification, Firebase Admin SDK for phone authentication integration, Razorpay payment webhook handling, Redis client libraries for distributed caching and rate limiting, and robust testing frameworks equivalent to pytest and vitest. If a proposed alternative lacks mature libraries for any of these, the evaluation must account for the engineering time required to build those integrations from scratch.

**Scaling requirements:** Based on the strategic plan's revenue targets, the platform should comfortably handle ten thousand monthly active users by month twelve, with peak concurrent usage during election season estimated at two to five thousand simultaneous users. API request patterns suggest each active user generates approximately fifty to one hundred API calls per session (loading state data, constituency lists, running multiple prediction scenarios, viewing bookmarks). This translates to peak load of roughly one hundred to two hundred requests per second during election-season spikes, with baseline load of five to ten requests per second between elections. The backend must handle this load with median latency below 500ms and p95 latency below two seconds, while database query performance must not degrade as the dataset grows from 574,000 rows to an estimated 800,000 rows over the next two years.

## Success Criteria

The tech stack evaluation is complete and actionable when it produces the following deliverables.

**Quantitative performance comparison:** A table comparing the current optimized Python/FastAPI stack against at least two alternatives (Go and Node.js backends), measuring throughput (requests per second), median latency, p95 latency, and p99 latency for the five most-used API endpoints (state list, year list for state, constituency detail, winner list for state-year, national party summary). These measurements should be based on either benchmarking tools like wrk or ab running against deployed instances, or extrapolation from published framework benchmarks for applications with similar database access patterns. The comparison must account for cold start time when using serverless or container-based hosting that suspends inactive instances.

**Cost model at three traffic levels:** A spreadsheet or table showing monthly infrastructure cost for baseline traffic (one thousand MAU), moderate growth (ten thousand MAU), and election-season peak (fifty thousand MAU over one week) for each stack option. Cost modeling must include compute costs for the backend API server (accounting for instance size, RAM, CPU, and auto-scaling), database hosting costs (managed PostgreSQL or equivalent), Redis or equivalent caching infrastructure, bandwidth costs for API responses and frontend static assets, monitoring and logging costs (assuming Sentry or equivalent for error tracking and Grafana Cloud or equivalent for metrics), and Firebase phone authentication costs based on projected SMS volume. The model must identify which cost component dominates at each traffic level.

**Migration effort estimate:** For any alternative stack, a timeline estimate in developer-weeks for implementing feature parity, including backend endpoint reimplementation, database access layer replacement (if applicable), authentication integration (Firebase + JWT), payment processing webhook handling (Razorpay), test coverage equivalent to the target eighty percent from the strategic plan, and deployment configuration for the chosen hosting platform. The estimate must include a risk assessment of what could go wrong during migration (data loss, authentication breakage, payment processing failures) and a rollback plan.

**Developer productivity assessment:** A qualitative evaluation of how each stack affects development velocity for common tasks: adding a new API endpoint, modifying a complex SQL query, adding a new test case, debugging a production error using logs and traces, implementing a new feature that touches backend and frontend, and deploying a hotfix during an incident. The assessment should consider factors like framework boilerplate requirements, type safety and compile-time error catching, testing tool maturity, and community ecosystem size.

**Horizontal scaling readiness:** An explicit statement of what changes are required for each stack option to scale horizontally across multiple backend instances. For the current Python/FastAPI stack, this means documenting the migration from in-memory rate limiting to Redis-backed rate limiting and confirming that session state is already externalized (JWT tokens are stateless). For alternative stacks, this means verifying that the same externalization is possible and identifying any framework-specific scaling gotchas.

**Technology-specific performance characteristics:** For each stack option, document the inherent performance characteristics relevant to this workload: connection pooling behavior for the database client library and whether it supports async/await or requires connection per request, JSON serialization performance for large result sets (some endpoints return hundreds of constituency records), memory efficiency for long-running processes (important for serverless or container-based hosts that charge by RAM-time), and cold start time for container-based deployments.

**Recommendation with explicit trade-offs:** A clear recommendation to either keep the current stack with specified optimizations or migrate to a specific alternative, with explicit trade-off documentation. The recommendation must state which scenario favors each option. For example: "Keep Python/FastAPI if developer productivity is the top priority and current performance is acceptable after Redis caching is enabled. Migrate to Go if p50 latency must drop below 100ms and infrastructure cost is highly constrained. Migrate to Node.js if the developer wants to unify the stack to a single language and TypeScript type safety across frontend and backend is valued."

**Open questions for escalation:** Any critical unknowns that cannot be resolved through desk research or synthetic benchmarking, such as: What is the current production p50 and p95 latency for the top ten endpoints, measured over a seven-day period during normal traffic? What is the current memory consumption of a Railway backend instance under baseline and peak load? What is the current Firebase authentication cost per month based on actual SMS volume? These questions require production instrumentation and should be flagged for implementation before making a final decision.

## Open Questions

The following ambiguities and gaps in the problem statement require resolution during the evaluation:

**What is the current baseline performance?** The strategic plan set targets of sub-500ms median latency and sub-2s p95 latency, but there are no production measurements confirming whether the current stack meets, exceeds, or fails these targets. If the current stack already meets the targets after Redis caching is enabled, then migration may not be necessary. If it fails by a small margin, optimization may suffice. If it fails by a large margin, migration may be warranted. The evaluation must either instrument production to gather this data or use synthetic benchmarks to estimate it.

**Is the concern performance, cost, or something else?** The prompt mentions "performance and scalability at the cheapest cost," but these are potentially conflicting goals — maximizing performance often increases cost (larger instances, more aggressive caching, edge deployment), while minimizing cost often degrades performance (smaller instances, cold starts, fewer cache layers). The evaluation must clarify which is the dominant constraint. If cost is paramount and current performance is acceptable, the question becomes "how can we reduce hosting costs while maintaining performance?" If performance is paramount and cost is secondary, the question becomes "what is the fastest stack regardless of cost?"

**Are there specific performance pain points observed in production?** Has the developer or users reported slow page loads, timeouts, or degraded experience during traffic spikes? If yes, which endpoints are slow? The evaluation should prioritize investigating those specific bottlenecks. If no pain points have been observed, this may indicate the current stack is adequate and migration is premature optimization.

**What is the developer's risk tolerance for migration?** A migration that improves performance by twenty percent but takes three months and risks data loss may not be worth it compared to spending those three months building monetization features. The evaluation must consider opportunity cost. If the recommendation is to migrate, it must justify why the performance/cost improvement is worth the delay to other roadmap items.

**Are there non-performance reasons to consider migration?** For example, does the developer want to consolidate to a single language (TypeScript for both frontend and backend) for conceptual simplicity, even if performance is equivalent? Does the developer want to learn Go or Rust for career development reasons? These are valid considerations but should be stated explicitly rather than framed as technical requirements.

**What does "scalability" mean in this context?** Does it mean the ability to handle 100x traffic spikes through auto-scaling, or does it mean the ability to grow from 574,000 rows to five million rows without degrading query performance, or does it mean the ability to expand to real-time features like live election result streaming? Each of these implies different architectural choices, and the evaluation cannot address all of them without prioritization.

**Refined Prompt Summary**

Evaluate whether the current Python FastAPI + React + PostgreSQL stack, deployed on Railway and Vercel free tiers, is the optimal technical foundation for an Indian election analytics platform targeting one thousand dollars MRR within twelve months, ten thousand monthly active users, and sub-500ms API latency. Compare against at least two alternatives (Go-based backend, Node.js-based backend) across performance benchmarks, cost models at three traffic levels, migration effort, developer productivity, and horizontal scaling readiness. Produce a recommendation with explicit trade-offs stating when to keep the current stack with optimizations versus when to migrate, and identify open questions requiring production instrumentation. The analysis must account for a solo developer context, extreme traffic cyclicality aligned with election schedules, and cost sensitivity requiring infrastructure spending below twenty percent of revenue.
