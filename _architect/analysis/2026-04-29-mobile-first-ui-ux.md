## 1. Introduction

The Indian election analytics platform serves a diverse audience — voters seeking constituency-level insight, political analysts building prediction models, journalists researching historical trends, and researchers studying electoral patterns across thirty-two years of data. The platform currently delivers its value through a React 19 frontend backed by a FastAPI Python API and PostgreSQL database, exposing interactive prediction sliders, historical charts, constituency tables, and a community prediction feed. While the desktop experience is competent with strong data visualization and a polished dark theme, the mobile experience — where an estimated forty to sixty percent of Indian internet users consume web content — suffers from a desktop-first design philosophy that has been retrofitted with breakpoint overrides rather than designed from the ground up for small screens.

This document presents a strategic analysis and implementation plan for transforming the platform into a mobile-first, ultra-modern, and highly accessible application. The transformation encompasses CSS architecture inversion, design system creation, touch interaction optimization, accessibility remediation, performance tuning, animation integration, and navigation redesign. The goal is not merely to make the existing UI "work" on mobile, but to reimagine the information architecture and interaction patterns so that mobile users experience the platform as a native-quality application purpose-built for their context — thumb-reachable controls, immediate tactile feedback, and progressive disclosure of the platform's considerable data density.

## 2. Motivation

A comprehensive Chrome DevTools audit of the current frontend reveals several concrete deficiencies that motivate this transformation. The Lighthouse Accessibility score stands at 82 out of 100, driven by two critical failures: four select elements lack associated labels or aria-label attributes, and the 381-row constituency table uses th elements without scope attributes or a caption element. The mobile header layout breaks catastrophically at 375px viewport width, with the title "Tamil Nadu Election Analysis" wrapping to four lines while action buttons float alongside it creating visual overlap. Fourteen interactive elements fall below the 44-by-44 pixel minimum touch target size recommended by both iOS Human Interface Guidelines and Material Design specifications, including the National button at 76-by-28 pixels, Assembly and Lok Sabha toggles at 85-by-25 pixels, and the Sign In button at 87-by-32 pixels. Zero ARIA attributes exist anywhere in the application — no roles, no aria-labels, no aria-describedby, no aria-labelledby — meaning screen reader users cannot meaningfully navigate the interface. Focus styles are actively suppressed with three focus rules that set outline to none without providing adequate visual replacement, rendering the application unusable for keyboard-only users.

The CSS architecture compounds these issues. All styling resides in a single 2000-line index.css file using eight CSS custom properties for theming. The responsive strategy defines base styles for desktop viewports (1400px max-width containers) and then applies override media queries at 900px, 600px, 380px, and 768px — the classic desktop-first approach that leads to mobile styles being patches on top of desktop layouts rather than a coherent small-screen design. The prediction panel, which is the platform's core interactive feature with anti-incumbency sliders, turnout controls, affinity weight grids for eight-plus parties, and constituency-level overrides for 234 seats, renders its full complexity on every screen size with only basic stacking adjustments, overwhelming mobile users with dozens of simultaneous controls that compete for limited viewport space.

Performance metrics, measured through DevTools, tell a more positive story: LCP at 227 milliseconds, CLS at 0.00, and TTFB at 14 milliseconds in development mode are all excellent. The bundle architecture with manual chunk splitting for Firebase, Recharts, and vendor code demonstrates good practices, and the estimated 165KB gzipped initial bundle is well within the 300KB budget. However, chart rendering on slider interaction currently triggers full recalculation of 234 constituencies and three chart re-renders on every onChange event without debouncing, which would cause perceptible lag on budget Android devices common in India. The Plausible analytics integration is broken due to an empty data-domain attribute in index.html, meaning no real-user data exists to inform mobile usage patterns.

The competitive landscape further motivates action. Platforms like FiveThirtyEight, The Economist's interactive graphics, and The Guardian's data desk have established expectations for mobile-first political data visualization that combine editorial design quality with touch-optimized interactions. Indian users accessing election data on mobile devices — whether during commutes, at polling stations, or in group discussions — deserve an experience that meets these standards rather than a scaled-down desktop interface.

## 3. Purpose

The precise objectives of this transformation are to invert the CSS architecture to a mobile-first paradigm where base styles target 320px viewports and progressively enhance through min-width media queries at 640px, 768px, 1024px, and 1280px breakpoints; to raise the Lighthouse Accessibility score from the current 82 to 100 by resolving all identified violations including unlabeled form elements, missing table scope attributes, absent ARIA landmarks, suppressed focus indicators, and inadequate keyboard navigation; to ensure all interactive elements meet or exceed 44-by-44 pixel touch targets with minimum 8px spacing between adjacent tappable areas; to establish a comprehensive design token system expanding the current eight CSS variables to approximately sixty tokens covering semantic color scales, typography scales, spacing, elevation, and corner radii; to implement a bottom navigation pattern for mobile that places primary routes within natural thumb-reach zones; to add purposeful micro-interactions and animations that provide immediate tactile feedback for slider adjustments, button presses, modal transitions, and chart data loading; to optimize chart rendering for touch devices with tap-to-reveal tooltips, repositioned legends, and debounced re-renders during slider interaction; to achieve Lighthouse Performance scores above 95 on read-only routes (landing page, national dashboard, state overview) and above 90 on interactive routes (prediction panel, constituency detail) on mobile-throttled audits while maintaining the current bundle size under 300KB gzipped; and to introduce progressive disclosure patterns in the prediction panel that present the most commonly used controls first while hiding advanced options behind expandable sections, reducing cognitive load on small screens without removing functionality.

Success will be measured against these criteria: Lighthouse scores above 95 for Performance on read-only routes (landing page, national dashboard, state overview) and above 90 on interactive routes (prediction panel, constituency detail), with 100 for Accessibility, 100 for Best Practices, and 95 or above for SEO across all core routes when measured in incognito mode with mobile throttling; browser support verified across Chrome 105+ and Android WebView 105+ (required for container queries), Safari 15.4+, and Firefox 103+; Core Web Vitals at the 75th percentile passing all thresholds with LCP under 2.5 seconds on production builds (not development mode — the current 227ms LCP baseline is a development-mode measurement not transferable to production mobile performance), INP under 200 milliseconds, and CLS below 0.1 on real mobile connections; zero axe-core accessibility violations on any page (verified via `@axe-core/playwright` integration, not only Lighthouse — since Lighthouse catches approximately 30-40% of WCAG violations while axe-core provides broader coverage); all animations maintaining 60fps on mid-range Android devices as verified by DevTools Performance profiling; and total JavaScript bundle under 300KB gzipped for initial page load. Note that the actual built bundle size must be measured via `vite build` before committing to all proposed dependency additions, as the 165KB estimate is based on dependency analysis rather than production build output.

## 4. Analysis

Three distinct approaches to this transformation were evaluated through deep technical analysis, competitive benchmarking, and assessment of the current codebase architecture.

### Approach A: Tailwind CSS 4.x Migration with Custom Components

This approach would migrate the existing 2000-line index.css to Tailwind CSS 4.x using utility classes applied directly to JSX elements, configured with a custom theme defining design tokens through Tailwind's at-theme directive. The mobile-first paradigm would be achieved naturally since Tailwind's responsive modifiers (sm, md, lg, xl) are min-width based by default, meaning unprefixed utilities target the smallest screens and prefixed utilities enhance for larger viewports. The existing eight CSS custom properties would be replaced by Tailwind's configuration file defining color scales (primary-50 through primary-900, neutral scales, semantic success/warning/error), typography scales using clamp-based fluid sizing, and spacing using a consistent 4px base unit. All 25 components would need className refactoring to replace CSS class names with Tailwind utilities. Animations would use Tailwind's built-in transition and animation utilities for simple effects, supplemented by the Motion library for complex gestures and layout animations.

The strengths of this approach for this specific codebase are significant. Tailwind 4.x includes a first-party Vite plugin that eliminates configuration complexity and enables instant Hot Module Replacement for utility class changes. The mobile-first modifier system directly addresses the core architectural inversion requirement without manual media query management. Container queries are supported as a core feature, enabling prediction panel components to adapt based on available space rather than viewport width — critical when the prediction sidebar transitions from a 320px-wide panel on desktop to full-width on mobile. Tree-shaking in Tailwind 4.x produces only the CSS for utilities actually used, typically resulting in 8-15KB gzipped CSS that would be smaller than the current 2000-line file. The utility-first approach eliminates naming collisions and specificity wars that plague large single-file CSS architectures.

The weaknesses are equally concrete. The migration effort requires touching every component's JSX to replace className strings, estimated at approximately 300 className changes across 25 components. Without an accessibility component library, all ARIA attributes, focus management, keyboard navigation, and screen reader support must be manually implemented — exactly the area where the current codebase has zero investment. The developer must maintain discipline to apply consistent accessibility patterns across every interactive element rather than relying on pre-built accessible primitives. Tailwind's utility approach can produce verbose JSX with long className strings that reduce code readability, particularly in data-dense components like PredictionConstituencyTable and ConstituencyList. Complex interactive components like modal dialogs, dropdown menus, and tab panels require custom implementation of WAI-ARIA patterns that component libraries provide for free.

### Approach B: Shadcn/ui with Radix UI Primitives and Tailwind

This approach combines Tailwind CSS as the styling engine with Radix UI as the accessibility primitive layer and Shadcn/ui as the component composition pattern. Radix UI provides unstyled, fully accessible headless components for dialogs, dropdown menus, sliders, tabs, toggle groups, select menus, and tooltips — each implementing complete WAI-ARIA patterns with keyboard navigation, focus management, and screen reader support built in. Shadcn/ui is not a traditional component library but rather a collection of copy-paste component implementations that use Radix primitives styled with Tailwind utilities, giving full ownership of the component code without dependency on a third-party component library's release cycle. The migration path would involve installing Radix primitive packages for each interactive pattern used in the codebase (dialog for modals, slider for prediction controls, tabs for navigation, select for dropdowns, toggle-group for election type selector), creating Shadcn-style wrapper components styled with Tailwind, and incrementally replacing current HTML-native implementations.

The strengths of this approach are compelling for the identified accessibility gaps. Every Radix primitive provides complete keyboard navigation, ARIA attributes, focus trapping, and screen reader announcements without any manual implementation — directly addressing the finding that zero ARIA attributes exist in the current codebase. The Slider primitive from Radix would replace the current native range input styling (which requires separate webkit and moz pseudo-element rules) with a fully accessible, touch-optimized slider that meets 44-by-44 pixel touch target requirements and supports both horizontal and vertical orientations. The Dialog primitive would replace the current modal-overlay implementation in LoginModal and SaveBookmarkModal with proper focus trapping, escape-to-close, aria-describedby, and return-focus-on-close behaviors. The Tabs primitive would replace the horizontal scrolling button navigation with proper role=tablist, role=tab, and role=tabpanel semantics, arrow key navigation between tabs, and automatic ARIA state management. The Select primitive would resolve the Lighthouse failure caused by the four unlabeled select elements by providing built-in label association and keyboard navigation. All of these come with zero runtime overhead for accessibility — the accessibility is structural, not bolted on.

The weaknesses center on migration complexity and bundle size. Installing Radix primitives adds approximately 40-60KB gzipped JavaScript to the bundle, though individual primitives can be imported selectively (dialog is roughly 8KB, slider 6KB, tabs 5KB, select 10KB). The migration requires not just className changes (as with Tailwind-only) but structural component rewrites — LoginModal must be restructured from a div with onClick handlers to a Radix Dialog with Trigger, Portal, Overlay, Content, Title, Description, and Close subcomponents. The prediction panel sliders must migrate from native input type=range to Radix Slider with Root, Track, Range, and Thumb subcomponents. The constituency table's clickable rows need assessment for whether Radix's selection primitives add value or overhead. Each component rewrite carries risk of behavioral regression, particularly in the prediction engine's tight coupling between slider onChange events and the generateBaseline and applyNewParty functions. The Shadcn/ui pattern of owning component code means the team is responsible for maintaining and updating these components rather than receiving upstream fixes from a library.

### Approach C: Enhanced Custom CSS with PostCSS and Manual Accessibility

This approach preserves the existing CSS architecture while enhancing it with PostCSS plugins for nesting, custom media queries, and autoprefixing, expanding the CSS custom property system to a comprehensive design token structure, and manually inverting the media query strategy from desktop-first to mobile-first within the same index.css file structure. Accessibility would be addressed through manual addition of ARIA attributes, focus management code, and keyboard event handlers to each component without adopting a headless component library. The design token expansion would define approximately sixty variables organized into semantic groups (color-primary-50 through color-primary-900, spacing-1 through spacing-16, font-size-xs through font-size-4xl, shadow-sm through shadow-xl, radius-sm through radius-xl) within the root selector, replacing the current eight variables.

The strength of this approach is minimal disruption. No new dependencies are added to the bundle — zero additional JavaScript, zero new build tool configuration. The existing component structure remains unchanged; only CSS properties and HTML attributes are modified. The learning curve is near zero since the team already understands the CSS architecture. Git history is preserved with clean diffs showing CSS property changes rather than wholesale component replacements. The media query inversion is a mechanical transformation: move current base styles into a max-width 1280px query, elevate current 600px query styles to the base level, and progressively add complexity at min-width breakpoints. The design token expansion can be done incrementally — start with the most-used tokens (color, spacing) and expand to typography and elevation as components are touched.

The weakness of this approach is that it does not solve the fundamental problems. Manual accessibility implementation requires the developer to correctly implement WAI-ARIA patterns for every interactive element — dialog focus trapping, tab panel keyboard navigation, slider aria-valuemin/max/now attributes, combobox patterns for searchable selects, live region announcements for dynamic content — and to maintain these implementations as components evolve. The current codebase has demonstrated zero investment in accessibility, and this approach relies entirely on developer discipline to add and maintain what a library like Radix provides structurally. A single 2000-line CSS file, even with PostCSS nesting, will continue to grow and become harder to maintain as components are added or modified. There is no component encapsulation — styles for the prediction panel, constituency table, modals, and navigation all coexist in a single namespace where naming collisions and specificity conflicts are managed through convention rather than tooling. The approach does not provide any of the DX improvements that Tailwind or Shadcn offer, such as design-time validation of spacing consistency, responsive utility shorthand, or integrated accessibility testing.

### Approach Comparison

Comparing the three approaches against consistent criteria reveals clear tradeoffs. On accessibility outcomes, Approach B (Shadcn/Radix) is strongest because accessibility is structural and automatic; Approach A (Tailwind-only) is weakest because it requires full manual implementation identical to Approach C but without the familiarity of the existing CSS patterns. On migration effort, Approach C is lowest (CSS-only changes), Approach A is moderate (className refactoring across all components), and Approach B is highest (structural component rewrites plus className refactoring). On bundle size impact, Approach C adds zero bytes, Approach A adds approximately 10-15KB CSS (likely smaller than current), and Approach B adds 40-60KB JavaScript for Radix primitives. On long-term maintainability, Approach B is strongest with encapsulated accessible components and utility-class styling, Approach A is moderate with utility styling but manual accessibility debt, and Approach C is weakest with a growing monolithic CSS file and manual accessibility maintenance burden. On design system quality, both A and B leverage Tailwind's token system for consistency enforcement, while C relies on developer discipline to use the right custom property among sixty options without tooling assistance.

## 5. Suggestions

All three approaches are viable and can be phased. Approach C (Enhanced Custom CSS) ranks third — while it has the lowest migration effort, it perpetuates the fundamental architectural problems (monolithic CSS, no accessibility primitives, no design system tooling) and creates ongoing maintenance debt. It should be explicitly rejected as a primary strategy because the accessibility gaps identified in the DevTools audit demand structural solutions, not manual patches.

Approach A (Tailwind-only) ranks second — it solves the CSS architecture problems elegantly with mobile-first utility classes, container queries, and design token integration, but it leaves the accessibility gap unaddressed. Given that the current Lighthouse Accessibility score is 82 with zero ARIA attributes anywhere in the application, adopting a styling framework without accessibility primitives would require the same manual effort as Approach C to reach the target score of 100.

Approach B (Shadcn/ui with Radix and Tailwind) ranks first and is the recommended approach. It uniquely addresses both the CSS architecture inversion and the accessibility remediation simultaneously. The additional 40-60KB JavaScript bundle size is justified by the fact that implementing equivalent accessibility manually would require significantly more application code with higher defect risk, and the current estimated 165KB gzipped bundle fits within the 300KB budget — though proposed additions of Radix (40-60KB), Motion (22.7KB), Inter font (17KB), and TanStack Virtual (12KB) total 92-112KB, leaving only 23-43KB of headroom, making actual bundle measurement via `vite build` essential before committing to all dependencies. The higher migration effort is justified by the higher quality ceiling — each migrated component gains correct accessibility behavior for the lifetime of the application rather than requiring ongoing accessibility audits and patches.

The approaches can be partially combined: the Shadcn/Radix approach inherently uses Tailwind for styling, so Approach B subsumes Approach A. Elements of Approach C (design token expansion) are incorporated into the Tailwind configuration rather than being a separate effort. The recommended strategy is therefore Approach B as the primary direction, incorporating Tailwind's token system (from A) and the incremental migration philosophy (from C).

## 6. Recommended Suggestion

The recommended approach is Approach B: Shadcn/ui with Radix UI Primitives and Tailwind CSS 4.x, executed as an incremental migration that preserves the existing component structure while progressively replacing CSS classes with Tailwind utilities and native HTML interactive elements with Radix primitives.

This selection is justified by three converging requirements. First, the DevTools audit revealed that accessibility is the single largest quality gap — a Lighthouse score of 82 with zero ARIA attributes, suppressed focus styles, missing form labels, inaccessible table markup, and 381 clickable rows without keyboard access. Radix primitives resolve all of these structurally, meaning accessibility is enforced by the component architecture rather than relying on developer memory. No other approach provides this guarantee. Second, the mobile-first CSS inversion requires a design system that enforces consistency across sixty-plus tokens, fluid typography, responsive spacing, and container-aware layouts. Tailwind 4.x with its at-theme directive and first-party Vite plugin provides exactly this with zero configuration overhead and tree-shaken output smaller than the current CSS file. Third, the Shadcn/ui composition pattern — where component source code is owned by the project rather than imported from a library — means the team retains full control over styling, behavior, and bundle optimization without being locked into a third-party component library's API or release cycle. This ownership is a double-edged sword: while it avoids dependency on external release cycles, it also means the team bears full responsibility for applying upstream Radix security patches and accessibility fixes that a library import would receive automatically. This tradeoff is acceptable for a project of this size where the number of owned components (7-8 primitives) is manageable, but it must be acknowledged as an ongoing maintenance commitment rather than presented as a pure advantage.

The strongest counterargument against this approach is the migration effort. Structural component rewrites for modals, sliders, tabs, selects, and navigation introduce risk of behavioral regression, particularly in the prediction engine's tight coupling between slider inputs and the generateBaseline/applyNewParty calculation pipeline. This risk is mitigated by the incremental migration strategy: each component is migrated, tested, and validated independently before the next begins. The existing Vitest and Testing Library setup, though currently containing only two test files (`predictionEngine.test.ts` and `api.test.ts`), provides the infrastructure for component-level regression tests that should be written before each migration. Additionally, while the Radix Slider primitive uses a different callback signature — `onValueChange` receives a `number[]` (to support multi-thumb sliders) rather than a `ChangeEvent` with `e.target.value` as a string — the integration requires a thin adapter that extracts `values[0]` and passes it as a number to the prediction engine. Since `App.tsx` owns `predParams` state (via `setPredParams` at line 265) and passes it to `PredictionPanel.jsx` as a prop, the adapter must be coordinated between both components. This is a contained type conversion, not a data flow restructuring.

Approach A (Tailwind-only) falls short because it would solve the styling problem while leaving the accessibility problem for manual implementation — effectively doubling the work since every component would need both className refactoring for Tailwind and manual ARIA/keyboard implementation. Approach C (Enhanced CSS) falls short because it addresses neither the styling scalability problem nor the accessibility problem at the architectural level, creating indefinite maintenance debt in both dimensions.

## 7. Full Implementation Plan

### Phase 0: Proof of Concept Spike — Validate Toolchain and Calibrate Effort

Before committing to the full migration, a time-boxed spike of one week validates all critical assumptions and calibrates effort estimates for subsequent phases. This phase exists because the plan proposes simultaneously learning and adopting three new technologies (Tailwind 4.x, Radix UI, Motion) while migrating a 2,300-line CSS codebase — a scope that demands validated assumptions rather than estimates based on documentation reading.

**Step 0.1: Measure the actual production bundle.** Run `vite build` and record the actual gzipped output sizes for all chunks (main, vendor, firebase, recharts). Compare against the estimated 165KB baseline. This measurement determines the real headroom within the 300KB budget and may influence which dependencies are adopted. This step has no dependencies.

**Step 0.2: Verify Tailwind 4.x and Vite 8 compatibility.** Install `@tailwindcss/vite` and create a minimal Tailwind configuration alongside the existing `index.css`. Verify that Hot Module Replacement works correctly, that the Tailwind Vite plugin does not conflict with existing PostCSS or CSS processing, and that the Tailwind CSS IntelliSense VS Code extension supports v4 syntax. This step has no dependencies.

**Step 0.3: Migrate LoginModal end-to-end as a reference implementation.** Take `LoginModal.jsx` through the complete migration path: replace its CSS classes with Tailwind utilities, wrap it in a Radix Dialog primitive with focus trapping and accessibility, add a Motion entrance animation, and write Playwright tests for the result. This single component exercises every technology in the stack and produces a reference implementation that guides all subsequent migrations. Place the new Shadcn-style Dialog primitive in `components/ui/dialog.tsx` to establish the file organization convention (Shadcn primitives in `components/ui/`, application components in `components/`). Convert the file from JSX to TSX during migration. This step depends on Steps 0.1 and 0.2.

**Step 0.4: Calibrate effort estimates.** Based on the LoginModal migration, estimate per-component effort for the remaining migrations. Update the phase effort estimates in this plan with measured values. If the spike reveals unexpected blockers (Tailwind 4.x incompatibility, Radix performance issues, excessive bundle impact), reassess the approach before committing to the full migration. This step depends on Step 0.3.

**Step 0.5: Inventory existing mobile CSS rules.** Systematically catalog all rules in the 600px media query block (lines 1494-1742, approximately 250 rules) and the 380px block (lines 1744-1779). For each rule, document what component it affects, what visual behavior it controls, and whether it maps to a Tailwind utility or requires custom CSS. This inventory prevents losing subtle mobile-specific adjustments that were refined through usage when they are replaced by Tailwind utilities during Phase 3. This step has no dependencies and can be parallelized with Steps 0.2-0.3.

**Estimated effort:** 5-7 developer-days. **Risk:** Low — the spike is fully reversible. **Exit criteria:** LoginModal renders identically in both old and new systems, Playwright accessibility tests pass, bundle size delta is measured and acceptable.

### Phase 1: Foundation — Tailwind CSS 4.x Integration and Design Token System

This phase establishes the styling infrastructure without changing any visible UI, enabling all subsequent phases to use the new design system. Estimated effort: 5-8 developer-days.

**Step 1.1: Install and configure Tailwind CSS 4.x with the Vite plugin.** Add the tailwindcss and the at-vite-plugin-tailwind packages to dev dependencies. Create a tailwind configuration using the at-theme directive in a new CSS entrypoint file that imports Tailwind's base, components, and utilities layers. Configure the Vite plugin in vite.config.js to process Tailwind utilities. Verify that both the existing index.css and the new Tailwind stylesheet are loaded, allowing dual-mode operation during migration.

**Step 1.2: Define the comprehensive design token system in Tailwind configuration.** Create color scales for primary (teal/cyan family, avoiding politically charged saffron/green/blue), neutral (refined dark mode palette with background lightened from the current 0f0f1a to a softer 121218 for reduced halation), surface (elevated card backgrounds), and semantic colors (success, warning, error, info). Define typography tokens with fluid clamp-based sizing for text-xs through text-4xl that scale automatically without media queries. Define spacing tokens on a 4px base unit from spacing-0.5 (2px) to spacing-16 (64px). Define elevation shadow tokens from shadow-sm to shadow-xl for card layering. Define border-radius tokens from rounded-sm (4px) to rounded-2xl (16px). This step depends on Step 1.1.

**Step 1.3: Create CSS utility bridge for incremental migration.** Map existing CSS custom properties (--bg, --bg2, --text, --text-muted, --accent, --border) to their Tailwind token equivalents so that both systems reference the same values during the migration period. This allows components to be migrated one at a time without visual inconsistency. Set a firm deadline for removing the bridge: the old `index.css` file must be fully replaced and deleted by the end of Phase 3. To enforce this, add an ESLint rule or CI check that flags any remaining references to the old CSS class names in JSX files after Phase 3 completion, preventing the dual CSS system from becoming permanent technical debt. This step depends on Step 1.2.

**Step 1.4: Configure dark mode and prefers-color-scheme detection.** Set up Tailwind's dark mode configuration using the class strategy, allowing future light mode toggle while defaulting to dark. Add a script in the HTML head that reads system preference and applies the appropriate class before first paint, preventing flash of wrong theme. This step depends on Step 1.2.

**Risk point:** Tailwind 4.x and the existing index.css may have conflicting base resets (Tailwind's Preflight normalizes all elements). Mitigation: configure Tailwind to skip Preflight during migration, enabling it only after all components have been migrated and the old CSS file is removed.

### Phase 2: Accessibility Foundation — Radix UI Primitives for Core Interactive Patterns

This phase addresses the most critical DevTools audit findings by replacing native HTML interactive elements with Radix primitives that provide structural accessibility.

**Step 2.1: Install Radix UI primitive packages.** Add radix-ui/react-dialog, radix-ui/react-slider, radix-ui/react-tabs, radix-ui/react-select, radix-ui/react-toggle-group, radix-ui/react-tooltip, and radix-ui/react-visually-hidden as production dependencies. This step has no dependencies.

**Step 2.2: Create Shadcn-style Dialog component and migrate modals.** Build a Dialog wrapper component in `components/ui/dialog.tsx` using Radix Dialog primitives (Root, Trigger, Portal, Overlay, Content, Title, Description, Close) styled with Tailwind utilities. All Shadcn-style UI primitives should be placed in `components/ui/` to separate them from application-level components in `components/`, following Shadcn's established convention. If using the Shadcn CLI, configure `components.json` with the correct path aliases; alternatively, manually create components in the Shadcn style without the CLI, which avoids the need for `@/` path alias configuration in `vite.config.js`. Migrate LoginModal.jsx to use this Dialog component (building on the reference implementation from Phase 0 Step 0.3), preserving all Firebase authentication logic while gaining focus trapping, escape-to-close, aria-describedby, and return-focus-on-close behaviors. Migrate SaveBookmarkModal.jsx similarly — note that `SaveBookmarkModal` is currently imported eagerly in `App.tsx` (line 18), not lazy-loaded. When migrating it to use Radix Dialog, either convert it to a lazy import (`React.lazy(() => import('./components/SaveBookmarkModal'))`) to avoid adding Radix Dialog's ~8KB to the main bundle, or accept the 8KB initial bundle increase and document it in the bundle budget. The lazy import approach is recommended. Write Playwright component tests for each modal verifying accessibility behaviors. This step depends on Steps 1.1 and 2.1.

**Step 2.3: Create Shadcn-style Slider component and migrate prediction controls.** Build a Slider wrapper in `components/ui/slider.tsx` using Radix Slider primitives (Root, Track, Range, Thumb) styled with Tailwind. Ensure thumb touch targets are 44-by-44 pixels minimum with `touch-action: none` on the slider container to prevent browser pan/zoom gestures from interfering with slider interaction (this is scoped to slider containers only — browser zoom must remain functional on the rest of the page for accessibility). Implement an adapter layer to handle the Radix Slider type mismatch: Radix's `onValueChange` callback receives a `number[]` (since it supports multi-thumb sliders), while the current prediction engine reads `e.target.value` as a string. The adapter should extract `values[0]` from the array and pass it directly as a number. Since `App.tsx` owns the `predParams` state and passes `setPredParams` to `PredictionPanel.jsx` via an onChange prop, both components must be updated: the Slider wrapper normalizes the value format, and `PredictionPanel.jsx` passes the normalized value upstream. Apply debouncing to the `setPredParams` calls that trigger chart-heavy re-renders (200ms delay) rather than debouncing chart rendering itself — because the existing `useMemo` in `App.tsx` (lines 129-155) already gates chart re-renders on dependency changes, the real issue is that every `setPredParams` call during a slider drag creates a new state object triggering `useMemo` recomputation. Numeric value displays should update immediately via local component state. Migrate all range inputs in PredictionPanel.jsx, converting to TSX during migration. Write Playwright component tests verifying slider accessibility (aria-valuemin, aria-valuemax, aria-valuenow, keyboard arrow key adjustment). This step depends on Steps 1.1 and 2.1.

**Step 2.4: Create Shadcn-style Tabs component and migrate navigation.** Build a Tabs wrapper using Radix Tabs primitives (Root, List, Trigger, Content) styled with Tailwind. Migrate the header navigation from the current horizontal scrolling button group to accessible tabs with role=tablist semantics, arrow key navigation between tabs, and proper aria-selected state management. On mobile, style the tab list as a bottom navigation bar positioned fixed at the bottom of the viewport with primary routes (National, Predictions, Community, Profile). This step depends on Steps 1.1 and 2.1.

**Step 2.5: Create Shadcn-style Select component and migrate dropdowns.** Build a Select wrapper using Radix Select primitives styled with Tailwind. Migrate the four unlabeled select elements identified in the Lighthouse audit (state selector, region filter, district filter, sort order), ensuring each has a properly associated label. This directly resolves the first Lighthouse accessibility failure. This step depends on Steps 1.1 and 2.1.

**Step 2.6: Add global accessibility infrastructure.** Add a skip navigation link as the first focusable element in the page. Add ARIA landmarks (role=banner for header, role=main for content, role=navigation for nav, role=contentinfo for footer). Add scope attributes to all th elements and caption to the constituency table, resolving the second Lighthouse accessibility failure. Add visible focus-visible styles to all interactive elements using a consistent ring style (2px offset, primary color ring). These changes apply to App.tsx and the constituency table components. This step involves manual ARIA work, which may appear to contradict the plan's argument for structural accessibility over manual implementation. The distinction is important: Step 2.6 covers page-level landmarks and table semantics that no component library provides (Radix primitives handle component-level accessibility such as dialogs, sliders, and tabs, but global landmarks, skip links, and table scope attributes are structural HTML concerns outside the scope of any headless component library). This manual work is a one-time addition of approximately 15-20 lines of HTML attributes, not an ongoing maintenance burden. This step has no dependencies and can be parallelized with Steps 2.2 through 2.5.

**Risk point:** Migrating prediction sliders to Radix may change the event timing or value format, breaking the tight coupling with predictionEngine.ts. Mitigation: write integration tests that verify slider value changes produce identical prediction outputs before and after migration, comparing the generateBaseline results for a fixed set of parameter combinations.

### Phase 3: Mobile-First Layout Inversion

This phase inverts the CSS architecture from desktop-first to mobile-first, restructuring component layouts for small screens as the base case.

**Step 3.1: Invert the header layout for mobile-first.** Redesign the header to stack vertically on mobile: title row (with hamburger or simplified branding), then subtitle, then action buttons (National, State Analysis, election type toggle, Sign In) as a full-width row with 44px minimum height touch targets, then the tab navigation bar. At 768px and above, transition to the current horizontal layout with title and actions on the same row. This directly fixes the DevTools finding of the header breaking at 375px with four-line title wrapping. This step depends on Phase 1 completion.

**Step 3.2: Invert the prediction panel layout.** Redesign PredictionPanel.jsx for mobile-first with the following structure: on mobile (base styles), render as a full-width stacked layout with collapsible accordion sections — primary controls (anti-incumbency, turnout) always visible, secondary controls (new party configuration) in a collapsed section, advanced controls (affinity weights, constituency overrides) in a further collapsed section. Use Radix Collapsible or Accordion primitives for smooth expand/collapse animations. At 1024px and above, render as the current fixed-width sidebar. This step depends on Phase 2 completion.

**Step 3.3: Invert data table layouts.** Redesign the constituency table for mobile-first: on mobile, show a card-based list view with constituency name, latest winner, party color indicator, and margin — tappable to expand full history. This card layout requires a brief design phase before implementation: sketch card variants for constituencies with varying data density (some with 10+ historical elections, some with 2-3) and validate that the card view surfaces sufficient summary information for quick scanning. At 768px and above, transition to the current horizontal-scroll table with sticky first column. This preserves the data density that analysts need on desktop while providing a scannable list for mobile users. Add scope attributes and caption per Step 2.6. This step depends on Phase 1 completion.

**Step 3.4: Invert chart layouts for mobile-first.** Configure all Recharts components to use mobile-optimized defaults: legends positioned below charts rather than to the right, responsive height using CSS viewport units on the parent container (e.g., `min-height: clamp(200px, 40vh, 400px)`) since Recharts' `ResponsiveContainer` only adapts width not height, reduced data density on mobile by showing the three most recent elections with a "Show all" toggle, and horizontal bar charts for party comparisons which are easier to read on narrow portrait screens. Add tap-to-reveal tooltips replacing the current hover-only Recharts tooltip that does not work on touch devices. This step depends on Phase 1 completion.

**Step 3.5: Implement bottom navigation for mobile.** Create a fixed bottom navigation bar visible on screens below 768px with four to five primary destinations: National (map icon), Predictions (slider icon), Community (users icon), and Profile/Bookmarks (bookmark icon). The bottom bar should use 56px height with 44-by-44 pixel touch targets, subtle elevation shadow above, and active state indicator (filled icon and accent color label). Hide the current top tab navigation on mobile when the bottom bar is active. This step depends on Step 2.4.

**Step 3.6: Implement the community feed mobile redesign.** Redesign CommunityFeed.jsx cards with elevated shadows, rounded corners (12px), clear visual separation, inline vote buttons styled as icon buttons with scale-on-press feedback, author avatar with initials fallback, and skeleton loading states during data fetch. Implement infinite scroll pagination using IntersectionObserver to replace any load-more button patterns. This step depends on Phase 1 completion.

**Risk point:** The card-based mobile table view (Step 3.3) may lose information that analysts depend on for quick scanning. Mitigation: user test both card and table views with political analyst personas to validate that the card view surfaces sufficient summary information; retain the table view as an accessible alternative behind a "Table view" toggle on mobile.

### Phase 4: Animation and Micro-Interaction Layer

This phase adds purposeful motion design that provides tactile feedback and improves perceived performance. Estimated effort: 5-7 developer-days.

**Step 4.0: Establish centralized animation configuration.** Before implementing individual animations, create a shared animation constants file (e.g., `lib/motion.ts`) defining Motion variants, transition presets, and duration tokens used across all components. This prevents animation definitions from being scattered as inline objects in individual components and ensures consistent timing, easing, and behavior across the application. Define presets for: page transitions (fade+slide, 200ms), interactive feedback (scale spring, 60ms), accordion expand/collapse (layout animation), and modal entrance/exit (fade overlay + slide content). This step depends on Step 4.1.

**Step 4.1: Install and configure the Motion library.** Add the motion package (formerly Framer Motion, 22.7KB gzipped) as a production dependency. The Motion library was selected over React Spring because it uses the Web Animations API for GPU-composited animations off the main thread, which is critical for maintaining 60fps on budget Android devices common in the Indian market. React Spring runs all animations on the main thread via requestAnimationFrame, which is problematic under CPU throttling. This step has no dependencies.

**Step 4.2: Implement page transition animations.** Wrap route content in Motion's AnimatePresence component to animate between routes with a subtle fade and slide-up transition (enter: opacity 0 to 1, y 16px to 0 over 200ms; exit: opacity 1 to 0 over 150ms). Use layout animations for tab content switching within the National Dashboard's four-tab view. This step depends on Step 4.1.

**Step 4.3: Implement interactive element feedback.** Add scale-on-press animations to all buttons (scale 0.97 on press with spring physics, 60ms duration). Add slider thumb scaling on touch (scale 1.2 on press with spring return). Add ripple-style feedback on vote buttons in the community feed. Add smooth height transitions for collapsible prediction panel sections. All animations should respect the prefers-reduced-motion media query, falling back to instant state changes for users with vestibular disorders. This step depends on Step 4.1.

**Step 4.4: Implement loading state animations.** Replace all loading spinners with skeleton screens that match the final content layout — shimmer effects using CSS gradients and animations for card placeholders, table row placeholders, and chart area placeholders. Implement staggered entrance animations for lists (community feed cards, constituency list items) with 50ms delay between items creating a cascading effect. This step depends on Step 4.1.

**Step 4.5: Implement chart entrance animations.** Add mount animations to Recharts components where chart bars grow from zero height and line charts draw progressively from left to right. Recharts supports the isAnimationActive prop with configurable duration; tune this for mobile (400ms duration, ease-out easing) to create polished data reveal effects. These entrance animations apply only on initial component mount, not on data updates triggered by slider changes — slider-driven chart updates must render instantly (or after the debounce period) without additional entrance animation, otherwise the total feedback time (200ms debounce + 400ms animation = 600ms) would exceed the 200ms INP target. Use a boolean flag (e.g., `hasAnimatedOnce`) to distinguish initial mount from subsequent re-renders. This step has no additional dependencies beyond Recharts configuration.

**Risk point:** Motion's `layout` animation uses FLIP-based transforms that can temporarily distort SVG content. If a Motion layout animation is applied to a container wrapping a Recharts `ResponsiveContainer`, the SVG chart may appear distorted during the animation because Recharts measures container dimensions synchronously and does not handle animated size changes. Mitigation: exclude all Recharts containers from Motion layout animations and use opacity-only transitions for chart containers. Apply layout animations only to non-chart containers (accordion sections, card lists, modal content).

**Risk point:** Animations may cause frame drops on budget Android devices even with Motion's Web Animations API approach. Mitigation: profile all animations on a CPU-throttled 4x environment in DevTools Performance panel; disable or simplify any animation that drops below 50fps; the prefers-reduced-motion respect provides an automatic fallback.

### Phase 5: Performance Optimization and Validation

This phase ensures the redesigned application meets all performance targets on mobile devices and Indian network conditions. Estimated effort: 5-8 developer-days.

**Step 5.0: Measure and validate bundle budget.** Run `vite build` and record actual gzipped sizes for all chunks. Compare against the pre-migration baseline from Phase 0 Step 0.1. Verify total initial JavaScript remains under 300KB gzipped. If budget is exceeded, identify which lazy-loaded dependencies have leaked into the main bundle (common with Radix primitives used in eagerly-imported components) and restructure imports. This step has no dependencies.

**Step 5.1: Implement chart render debouncing.** Wrap chart component re-renders in a 150-200ms debounce during slider interactions so that the prediction engine's generateBaseline and applyNewParty functions still execute immediately (keeping numeric displays responsive) but chart SVG re-renders are batched. Use React.memo with custom comparison functions on chart wrapper components to prevent re-renders when chart data has not meaningfully changed. This step has no dependencies.

**Step 5.2: Virtualize long lists.** Integrate TanStack Virtual (formerly react-virtual) for the constituency table on desktop view (381 rows) and the community feed if it grows beyond 50 items. Render only visible rows plus a small overscan buffer, reducing DOM node count from hundreds to approximately twenty visible rows. This step has no dependencies.

**Step 5.3: Optimize bundle loading.** Defer Firebase SDK initialization to after first meaningful paint — currently Firebase is loaded as a manual chunk but initialized synchronously. Lazy-load the Firebase Auth module only when the user taps Sign In. Preload critical route chunks for the most likely navigation paths (landing to national dashboard to state overview). Add font preloading for the selected web font using link rel=preload in the HTML head. This step has no dependencies and can be parallelized.

**Step 5.4: Run comprehensive Lighthouse audits.** Execute Lighthouse audits in Chrome DevTools incognito mode with mobile throttling (Moto G4, 4x CPU, Slow 4G) on all five critical routes. Document scores for Performance, Accessibility, Best Practices, and SEO. Identify and resolve any regressions from the redesign. Target: 95+ Performance on read-only routes (landing, national dashboard, state overview), 90+ on interactive routes (prediction panel, constituency detail), 100 Accessibility, 100 Best Practices, 95+ SEO. This step depends on completion of Phases 1 through 4.

**Step 5.5: Profile animations on throttled devices.** Use DevTools Performance panel with 4x CPU throttling to profile all animations during typical user flows: navigating between routes, adjusting prediction sliders, opening and closing modals, scrolling community feed, and expanding accordion sections. Identify any frame drops below 50fps and simplify or disable affected animations. This step depends on completion of Phase 4.

**Step 5.6: Fix Plausible analytics integration and update CSP.** Set the data-domain attribute in `index.html` to the production domain (this value is environment-specific and should be injected at build time or deployment). Integrate web-vitals library reporting to send LCP, INP, and CLS metrics to the analytics endpoint, segmented by device type and connection speed using the Network Information API. Update the Content-Security-Policy header in `api/main.py` (line 250) to add the following allowances: add `https://plausible.io` to `script-src` for the Plausible analytics script, add `https://plausible.io` to `connect-src` for analytics event reporting, and add a web-vitals reporting endpoint to `connect-src` if metrics are sent to a separate analytics service. For web fonts, self-host Inter and Noto Sans Devanagari as woff2 files bundled in the build output rather than loading from Google Fonts CDN — this avoids the need to add `font-src` CSP directives for external origins and ensures fonts work under the existing `default-src 'self'` policy. Regarding version pinning: all new dependencies added by this migration must use exact versions in `package.json` (no `^` ranges). Pinning existing dependencies (which currently use `^` ranges) is recommended but is out of scope for this migration and should be addressed as a separate maintenance task. This step has no dependencies.

**Step 5.8: Verify browser support matrix.** Test the completed application against the defined browser support matrix: Chrome 105+ and Android WebView 105+ (minimum for container queries), Safari 15.4+ (minimum for CSS `@layer` and `:has()`), and Firefox 103+ (minimum for container queries). Verify that all CSS features used (container queries, `clamp()`, `@layer`, CSS nesting) work correctly in these minimum versions. Test on at least one budget Android device running Chrome/WebView 105 to validate that container queries and Tailwind 4.x output are compatible. If any feature is unsupported in the minimum browser versions, add a PostCSS fallback or adjust the feature to use a supported alternative. This step depends on completion of Phases 1 through 4.

**Step 5.7: Conduct cross-device testing.** Test on physical devices or accurate emulators covering the primary Indian mobile market: a budget Android device (Samsung Galaxy A series or equivalent with 2-3GB RAM), a mid-range Android (Pixel 7 or Samsung Galaxy S21), and an iPhone (14 or SE). Verify touch interactions, animation smoothness, layout correctness, and overall experience quality. Document findings and address any device-specific issues. This step depends on completion of Phases 1 through 4.

### Phase Dependencies and Parallelization

Phase 0 (PoC Spike) must complete before any other phase begins, as it validates toolchain assumptions and calibrates effort estimates. Phase 1 (Foundation) must complete before Phases 2-5, as all subsequent work depends on Tailwind configuration and design tokens. Phase 2 (Accessibility) and Phase 3 (Mobile Layout) can proceed in partial parallel — Steps 2.6 (global accessibility infrastructure) and Steps 3.1, 3.3, 3.4, 3.6 (layout inversions that do not depend on Radix components) can run concurrently with Steps 2.2-2.5 (Radix component migrations). Steps 3.2 and 3.5 depend on Phase 2 completion because they use Radix Accordion/Collapsible and Radix Tabs respectively. Phase 4 (Animation) can begin as soon as its target components exist in their new form, so it can overlap with late Phase 3 work. Phase 5 (Performance) is primarily a validation phase that depends on Phases 1 through 4 being substantially complete, though Steps 5.0, 5.1, 5.2, and 5.3 (optimization changes) can begin independently at any time.

### JSX to TSX Migration Strategy

The codebase uses a mix of `.jsx` and `.tsx` files. Rather than conducting a separate JSX-to-TSX conversion effort, each component should be converted from `.jsx` to `.tsx` during its migration in Phases 2-3. When a component is touched for Tailwind utility migration or Radix primitive integration, convert the file extension and add TypeScript types at the same time. This avoids a separate migration phase while ensuring that all migrated components benefit from TypeScript's type safety, particularly for Radix component props which have well-defined TypeScript interfaces.

### Ordering Rationale

The phases are ordered to deliver value incrementally while managing risk front-to-back. Phase 0 validates assumptions before committing resources. Phase 1 establishes infrastructure without visible changes, creating the foundation for everything else. Phase 2 addresses the most critical quality gap (accessibility) first, because accessibility violations represent legal and ethical obligations that should not be deferred. Phase 3 delivers the primary user-visible transformation (mobile-first layouts) building on the accessible components from Phase 2. Phase 4 adds polish and delight through motion design, which should come after the structural layout is correct. Phase 5 validates and optimizes the complete system, which can only be meaningfully done after all changes are in place. Each phase produces a testable, deployable increment — Phase 0 produces a reference implementation and validated toolchain, Phase 1 can be deployed with zero user impact, Phase 2 improves accessibility immediately, Phase 3 transforms the mobile experience, Phase 4 adds polish, and Phase 5 ensures performance targets are met.

### Total Effort Estimate

For a solo developer who has not previously used Tailwind 4.x, Radix UI, or Motion, the total estimated effort is 10-16 weeks of focused work, broken down as: Phase 0 (1 week), Phase 1 (1-1.5 weeks), Phase 2 (2-3 weeks), Phase 3 (3-4 weeks), Phase 4 (1-1.5 weeks), Phase 5 (1-2 weeks), plus 1-2 weeks of buffer for unexpected issues. This estimate includes learning curve time (concentrated in Phase 0), test writing (approximately 30% of total effort), and JSX-to-TSX conversion (done incrementally per-component). The estimate assumes full-time dedicated effort; if interleaved with feature development, the calendar time will be proportionally longer. The effort can be reduced by approximately 20% if the developer has prior Tailwind or Radix experience.

## 9. Full Test Plan

### Testing Infrastructure

The test infrastructure requires two runners: Vitest with Testing Library for unit and component tests (already configured in the project), and Playwright for end-to-end tests, visual regression tests, and accessibility audits. Playwright must be installed and configured during Phase 0 as part of the PoC spike, since it is needed for the LoginModal reference implementation tests. The Playwright configuration should target Chromium, Firefox, and WebKit to match the browser support matrix, with mobile viewport presets for 375px (iPhone SE), 390px (iPhone 14), and 412px (Pixel 7). Install `@axe-core/playwright` for automated accessibility testing that goes beyond Lighthouse's 30-40% WCAG coverage — this integration runs axe-core assertions within Playwright tests, catching violations that Lighthouse misses such as incorrect ARIA role usage, missing accessible names on custom components, and color contrast failures in dynamic states.

### Unit Tests

Unit tests should cover each newly created Shadcn-style component in isolation. The Dialog component requires tests verifying that it renders in a portal element when triggered, traps focus within the dialog content preventing tab from escaping to background elements, returns focus to the trigger element when closed, closes when the Escape key is pressed, and exposes aria-describedby linking to the description element. The Slider component requires tests verifying that it renders with correct aria-valuemin, aria-valuemax, and aria-valuenow attributes reflecting the current parameter value, responds to arrow key presses by incrementing and decrementing the value by the configured step, calls the onChange handler with the correct numeric value, and renders a thumb element meeting the 44-by-44 pixel minimum touch target size. The Tabs component requires tests verifying that it renders with role=tablist on the container and role=tab on each trigger, manages aria-selected state correctly when tabs change, supports arrow key navigation between tab triggers, and renders the correct tabpanel content for the active tab. The Select component requires tests verifying that it renders with a properly associated label (resolving the Lighthouse failure), opens a popover of options on click or Enter key, supports keyboard navigation through options with arrow keys, and announces the selected option to screen readers.

For existing components that are refactored, regression tests should verify unchanged behavior. The prediction engine integration tests should confirm that for a fixed set of input parameters (anti-incumbency at 50 percent, turnout at 65 percent, growth factor at 1.05), the generateBaseline function produces identical prediction results regardless of whether the slider input is a native range element or a Radix Slider primitive. The LoginModal tests should verify that the Firebase phone authentication flow (country code selection, phone number submission, OTP entry, verification) works identically through the new Dialog component. The CommunityFeed tests should verify that vote interactions (like, dislike) produce the correct API calls and optimistic UI updates in the redesigned card layout.

### CSS Bridge Tests

During the migration period (Phases 1-3), tests should verify the CSS bridge mechanism works correctly. For each design token mapped in Step 1.3, verify that the old CSS custom property (e.g., `var(--bg)`) and the corresponding Tailwind token resolve to identical computed values. Write a Playwright test that renders a reference component under both styling systems and compares computed styles (background-color, color, font-size, padding) to verify visual equivalence. This test should run in CI on every pull request during the migration period and be removed when the old `index.css` is deleted at the end of Phase 3.

### Integration Tests

Integration tests should verify cross-component interactions that span the migration boundaries. The prediction flow integration test should simulate a user opening the prediction panel on mobile (bottom nav tap), adjusting the anti-incumbency slider from 0 to 75 using keyboard arrow keys (verifying Radix Slider accessibility), verifying that the prediction results summary card updates with the new seat distribution, verifying that the chart re-renders after the debounce period (200ms) with the correct data, and verifying that saving the configuration as a bookmark through the Dialog-wrapped SaveBookmarkModal produces the correct API request.

The navigation integration test should verify that on a 375px viewport, the bottom navigation bar is visible and the top tab bar is hidden, that tapping each bottom nav destination routes to the correct page with the page transition animation, that the active state indicator updates correctly on the bottom bar, and that on a 1024px viewport, the top tab navigation is visible and the bottom bar is hidden.

The accessibility integration test should verify that a keyboard-only user can navigate from the skip link to the main content, through the tab navigation using arrow keys, into the prediction panel sliders using Tab and adjusting values with arrow keys, through the community feed voting with Enter key activation, and back to the top via the skip link — completing a full user journey without requiring a mouse or touch input.

### End-to-End Tests

End-to-end tests should be implemented using Playwright and should exercise complete user workflows across the redesigned interface. The mobile prediction workflow should simulate a user on a 390px viewport landing on the national dashboard, navigating to a state via the map or bottom navigation, opening the prediction panel, adjusting primary controls (anti-incumbency and turnout sliders visible in the default expanded section), expanding the advanced controls section (collapsible accordion), modifying affinity weights, viewing the updated prediction results, and saving the prediction as a bookmark. The test should verify that all interactions are possible via touch-sized targets and that the final bookmark contains the correct parameter configuration.

The authentication workflow should simulate a user tapping Sign In on mobile, verifying the Dialog opens with proper focus management (focus moves to the first input, not the close button), entering a phone number with the numeric keyboard, receiving and entering the OTP in the six-box input (testing auto-tab between boxes), completing authentication, and verifying the dialog closes with focus returning to the Sign In button location (now showing the UserMenu). The test should run on both mobile and desktop viewports.

### Regression Tests

Every component that is migrated from CSS classes to Tailwind utilities or from native HTML to Radix primitives must have before-and-after visual regression tests using Playwright's screenshot comparison API. These tests capture screenshots of the component at 375px, 768px, and 1280px viewports before migration, then compare against screenshots after migration using a pixel difference tolerance of 0.1% to account for sub-pixel rendering differences and font anti-aliasing variations across environments. For data-driven components (Recharts charts), use structural comparison (verify data values rendered in SVG text elements) rather than pixel comparison, since floating-point calculations and text rendering produce slight visual differences that trigger false positives in screenshot comparison. The prediction engine output must be verified as byte-identical before and after the slider migration — any change in prediction results for the same input parameters represents a critical regression.

### Edge Cases and Boundary Conditions

Edge cases for the mobile layout include extremely narrow viewports (280px, representing older budget phones) where the bottom navigation icons must still be tappable without overlapping; extremely long constituency names that must truncate gracefully in both card and table views; right-to-left text rendering if any constituency or party names include Urdu script; the prediction panel with all sections expanded simultaneously on a 375px viewport (must scroll without layout breaking); and rapid slider adjustment (swiping a slider quickly from 0 to 100) must not cause JavaScript errors or visual glitches even with chart render debouncing.

Edge cases for the accessibility layer include screen reader announcement of dynamically updated prediction results (aria-live regions must announce new seat counts without overwhelming the user with constant updates during slider drag — announcements should be debounced to fire only after the user pauses); focus management when a modal is open and the viewport rotates from portrait to landscape; and keyboard navigation in the constituency table card view where each card must be focusable with Enter to expand and Escape to collapse.

### Negative Tests

Negative tests should verify that the system correctly handles invalid states. The prediction slider should not accept values outside its defined range (0-100 for percentage sliders) — Radix Slider enforces this structurally, but tests should confirm the UI cannot be manipulated to produce out-of-range values. The login modal should display clear inline error messages for invalid phone numbers (too few or too many digits) with the error text associated to the input via aria-describedby for screen reader users. The bookmark save modal should prevent submission with an empty title, displaying validation feedback that is both visible and announced to screen readers.

### Performance Tests

Performance tests should verify that the Lighthouse Performance score remains above 95 on mobile-throttled audits for the landing page, national dashboard, and state overview routes, and above 90 for the prediction panel and constituency detail routes (the lower target for interactive routes reflects the inherent cost of client-side prediction engine computation and Radix primitive hydration). Core Web Vitals tests should verify LCP under 2.5 seconds on production builds with mobile throttling (not development mode), INP under 200 milliseconds during slider interaction (note: this requires that chart entrance animations fire only on mount and that debouncing is applied to `setPredParams` rather than chart rendering), and CLS below 0.1 across all routes. Animation frame rate tests should verify that no animation drops below 50fps under 4x CPU throttling in DevTools, measured across page transitions, slider interactions, modal open/close, and community feed scrolling. Bundle size tests should verify that the total initial JavaScript bundle remains under 300KB gzipped and that no single lazy-loaded chunk exceeds 100KB gzipped. These measurements should be taken from `vite build` output, not estimated from dependency sizes.

### SEO Validation Tests

The mobile-first transformation introduces several SEO-relevant changes that must be validated. Content hidden behind accordion/collapsible sections (prediction panel advanced controls, constituency detail expanded history) should be verified as crawlable by Google's mobile-first indexer — use the URL Inspection tool in Google Search Console to verify that collapsed content is present in the rendered HTML. Bottom navigation should not change URL patterns from the current routing structure; existing bookmarked and shared URLs (e.g., `/state/Tamil%20Nadu/predictions`) must continue to work without redirects. The `robots.txt` and meta tags should be verified after the mobile layout changes to ensure no content is inadvertently marked as non-indexable.

### Manual Accessibility Testing

Automated testing (Lighthouse + axe-core) catches approximately 50-60% of WCAG 2.1 AA violations. The remaining violations require manual testing with assistive technologies. After Phase 2 completion, conduct a manual accessibility audit covering: (1) VoiceOver on macOS/iOS — navigate the complete prediction workflow using only VoiceOver, verifying that all interactive elements are announced with correct roles and states, that slider value changes are announced, and that modal focus trapping works correctly; (2) keyboard-only navigation — complete the full user journey (landing page to state overview to predictions to save bookmark) using only Tab, Shift+Tab, Enter, Escape, and arrow keys, verifying that focus order is logical, that all interactive elements are reachable, and that focus is never trapped outside of intentional traps (modals); (3) high-contrast mode — verify the interface remains usable with `prefers-contrast: more` media query active, ensuring focus indicators and interactive element boundaries remain visible. Document findings in a structured accessibility audit report.

## 10. How to Execute and Document the Implementation

Work should be structured into one pull request per phase, with Phase 0 as a single PR, Phase 2 potentially split into two pull requests (Steps 2.1-2.3 for Dialog and Slider migrations, Steps 2.4-2.6 for Tabs, Select, and global accessibility) due to its scope, and Phase 3 potentially split by component group. Each pull request should include a description referencing the specific phase and step numbers from this plan, screenshots or screen recordings showing the before and after state on mobile and desktop viewports, Lighthouse score comparisons (before and after), axe-core test results confirming zero new violations, and a checklist of acceptance criteria from the relevant phase.

Code review should focus on accessibility correctness (verify ARIA attributes are present and correct), responsive behavior at all breakpoints (reviewer should test in DevTools responsive mode at 375px, 768px, and 1280px), animation smoothness (reviewer should enable CPU throttling in DevTools and verify no frame drops), and design token usage consistency (verify that hardcoded color or spacing values are not introduced — all visual properties should reference Tailwind tokens).

Progress should be tracked through the todo list system with each phase and step represented as individual items. After each phase is complete, a brief status update should document the Lighthouse scores achieved, any deviations from the plan, and any issues discovered that affect subsequent phases. Rollback procedures vary by phase and must account for the CSS bridge complexity. Phase 0 is fully reversible — simply delete the PoC branch. Phase 1 is low-risk because it adds infrastructure without modifying existing code. Phase 2 rollback is more complex: if Phase 2 is partially completed (e.g., Dialog and Slider are migrated but Tabs and Select are not), rolling back requires reverting the structural component rewrites while preserving the Tailwind foundation. To manage this, each component migration within Phase 2 should be a separate commit (or set of commits) within the PR, enabling selective revert of individual component migrations without reverting the entire phase. ## 11. How to Execute and Document the Tests

Tests should be written before or during each component migration, following a test-driven approach for the accessibility behaviors. For each Radix component migration, the developer should first write Playwright tests specifying the expected ARIA attributes, keyboard behaviors, and focus management patterns, then implement the component to satisfy those tests. This ensures that accessibility is the driving requirement rather than an afterthought. Playwright should be installed and configured during Phase 0 (Step 0.3) alongside the LoginModal reference implementation — this validates the test infrastructure early and produces a reference test that guides subsequent component tests.

Test execution order should follow the phase structure: Phase 0 spike tests (Playwright setup verification, LoginModal accessibility tests, bundle measurement scripts), Phase 1 foundation tests (Tailwind rendering correctly alongside existing CSS, design tokens resolving to correct values, CSS bridge equivalence tests), Phase 2 accessibility tests (each Radix component passing individual and integration accessibility tests via Playwright + axe-core), Phase 3 layout tests (responsive behavior verified at all breakpoints through Playwright visual regression with 0.1% pixel tolerance), Phase 4 animation tests (frame rate assertions under throttled conditions), Phase 5 performance tests (Lighthouse score gates and bundle size assertions from `vite build` output).

Test results should be captured in the CI pipeline output and summarized in each pull request description. The test suite passes when all unit, integration, and accessibility tests return green, Playwright visual regression tests show no unintended differences beyond the 0.1% pixel tolerance threshold, axe-core reports zero violations on all tested pages, Lighthouse scores meet the defined thresholds (95+/90+ for performance, 100 for accessibility), and bundle size remains within budget as measured by `vite build` output. Test failures discovered during execution should be triaged as either migration regressions (fix immediately before proceeding) or pre-existing issues (document and create separate issues to address outside this migration scope).

Test coverage should be measured using Vitest's coverage-v8 provider for unit/component tests, with a target of 80 percent line coverage for newly created Shadcn-style components in `components/ui/` and 60 percent for migrated existing components. Playwright test coverage is measured by scenario completeness rather than line coverage — every user-facing workflow described in the end-to-end tests section must have a corresponding Playwright test. Coverage reports should be generated on each pull request and the coverage trend tracked across phases to ensure it increases rather than decreases as the migration proceeds. Given the current near-zero test coverage (only two test files exist), reaching these targets represents a substantial investment that is factored into the 10-16 week effort estimate.

## 12. Full Document Summary

This document presents a strategic plan for transforming the Indian election analytics platform from a desktop-first design with mobile breakpoint patches to a mobile-first, accessibility-first application using Shadcn/ui component patterns with Radix UI primitives for structural accessibility and Tailwind CSS 4.x for utility-first mobile-first styling. The transformation is motivated by concrete DevTools audit findings: a Lighthouse Accessibility score of 82 with zero ARIA attributes, a mobile header that breaks at 375px, fourteen touch targets below the 44-pixel minimum, and suppressed focus indicators that prevent keyboard navigation. Three approaches were analyzed — Tailwind-only, Shadcn/Radix with Tailwind, and enhanced custom CSS — with the Shadcn/Radix approach selected because it uniquely addresses both the CSS architecture inversion and the accessibility gap simultaneously through structural accessible primitives rather than manual ARIA implementation.

The implementation is structured in six phases: a proof-of-concept spike (Phase 0) that validates the toolchain, measures the actual bundle size, and produces a LoginModal reference implementation before committing resources; foundation (Phase 1, Tailwind and design tokens with a firm CSS bridge removal deadline); accessibility (Phase 2, Radix primitive migration with explicit adapter patterns for Radix Slider's `number[]` callback and lazy-loading of SaveBookmarkModal); mobile-first layout inversion (Phase 3, with design iteration required for card-based table views); animation layer (Phase 4, with centralized animation configuration and Recharts containers excluded from layout animations); and performance validation (Phase 5, including CSP header updates for Plausible/web-vitals, self-hosted web fonts, and browser support matrix verification against Chrome 105+, Safari 15.4+, Firefox 103+).

The total estimated effort is 10-16 weeks for a solo developer, including learning curve (concentrated in Phase 0), test writing (approximately 30% of effort using Playwright for E2E/visual regression and `@axe-core/playwright` for accessibility), and per-component JSX-to-TSX conversion. Key risks include Tailwind 4.x ecosystem maturity (mitigated by Phase 0 compatibility verification), Radix Slider type mismatch with the prediction engine (mitigated by a documented adapter pattern), CSS bridge permanence (mitigated by a firm removal deadline and lint rules), Recharts+Motion SVG distortion (mitigated by excluding chart containers from layout animations), and the tight bundle budget (measured 165KB estimate + 92-112KB proposed additions = 23-43KB headroom, mitigated by measuring actual build output in Phase 0).

Performance targets differentiate read-only routes (95+ Lighthouse Performance) from interactive routes (90+), with 100 for Accessibility, 100 for Best Practices, and 95+ for SEO. Success is further measured by zero axe-core violations (not just Lighthouse), LCP under 2.5 seconds on production builds, INP under 200 milliseconds, all touch targets at 44px minimum, all animations at 60fps on throttled devices, and bundle under 300KB gzipped. Shadcn component files are organized in `components/ui/` to separate UI primitives from application components. The Shadcn ownership model is acknowledged as both an advantage (no library lock-in) and a responsibility (manual upstream security/accessibility patch application). Manual accessibility testing with VoiceOver and keyboard-only navigation supplements automated testing to cover the 40-50% of WCAG violations that automated tools miss.
